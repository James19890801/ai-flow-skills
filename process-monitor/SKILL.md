# 流程智能监控设计 Skill

## 核心交付物

1. **《流程智能监控方案》**：含指标定义、阈值设置、异常分类、根因分析框架、优化决策树、Owner报告模板的完整结构化文档
2. **可视化监控看板（HTML文件）**：基于分析数据生成交互式HTML监控看板，包含实时指标展示、趋势图表、异常预警面板
3. **预警报告示例**：模拟一份Owner会收到的分析报告样本，帮助理解方案落地效果

## 鲁棒性保障规则（最高优先级）

> 用户提供信息不完整时，不能中断或报错，必须基于已有信息智能补全，并标注"AI推断"。

| 异常情况 | 处理策略 |
|----------|----------|
| 用户只说流程名，没有其他信息 | 基于流程名推断常见痛点和指标，生成草案，告知用户可以修正 |
| 关键节点未提供 | 基于流程类型（审批/运营/服务）推断3-5个典型节点，标注"AI推断，请确认" |
| 无 SLA 目标 | 使用行业基准值并标注来源，如"合同审批行业基准：3-5个工作日" |
| 用户描述模糊 | 不要求澄清所有信息，先出完整方案，在报告末尾列出"建议确认的问题" |
| 步骤1信息采集无响应 | 超过1轮无有效输入后，直接进入方案生成，用最保守的通用模板 |

## 执行流程

```
用户描述流程名称与业务目标
    ↓
【步骤 1】信息采集
    - 流程名称与所属业务域
    - 流程核心目标（如：合同审批 → 目标：3天内完成审批）
    - 关键节点（2-5个主要审批/处理节点）
    - 当前已知的痛点（可选）
    ↓
【步骤 2】指标设计
    - 推荐2-4个核心监控KPI
    - 每个指标含：名称 / 定义 / 计算方式 / 正常范围 / 异常阈值
    - 优先推荐：时效类指标 + 质量类指标 + 异常类指标
    ↓
【步骤 3】异常规则设置
    - 超时类异常：超过时限X天/小时触发预警
    - 高驳回异常：驳回率超过阈值触发分析
    - 重复异常：同一实例被驳回≥N次触发深度分析
    - 断点异常：流程中断/挂起超过X小时触发报警
    - 用户可自定义补充规则
    ↓
【步骤 4】根因分析框架
    - 针对每类异常，给出标准排查思路
    - 例：高超时率 → 排查：节点处理人负荷 / 信息不完整 / 流程规则问题
    ↓
【步骤 5】优化决策树
    根据异常类型，给出决策路径：
    - 改规则（调整审批条件/阈值）
    - 改流程（调整节点顺序/增减节点）
    - 授权（增加审批人/设置代理规则）
    - 培训（执行者能力问题 → 培训）
    - 自动化（高置信度场景 → 自动审批）
    ↓
【步骤 6】输出监控方案
    生成完整《流程智能监控方案》文档
    ↓
【步骤 7】生成可视化监控看板（HTML）
    基于分析数据生成交互式HTML监控看板，包含：
    - 核心指标卡片（平均时长、超时率、异常率、通过率）
    - 流程类型分布图表
    - 节点瓶颈分析可视化
    - 异常预警列表面板
    - 趋势分析图表
    ↓
【步骤 8】生成Owner预警报告模板
    基于方案，生成一份模拟的Owner预警通知示例
```

## 标准监控指标体系

### 时效类指标
| 指标 | 定义 | 建议阈值参考 |
|------|------|------------|
| 平均审批时长 | 从提交到最终完成的平均天数 | 根据SLA定义 |
| 超时率 | 超过约定时限的实例占比 | >20%触发预警 |
| 节点滞留时长 | 单个节点平均等待时间 | >SLA的50%触发预警 |

### 质量类指标
| 指标 | 定义 | 建议阈值参考 |
|------|------|------------|
| 驳回率 | 被驳回实例占总数比例 | >30%触发异常 |
| 重复驳回率 | 同一实例被驳回≥2次的比例 | >10%触发深度分析 |
| 一次通过率 | 首次提交即通过的比例 | <60%需关注 |

### 异常类指标
| 指标 | 定义 | 建议阈值参考 |
|------|------|------------|
| 周驳回频次 | 同一流程一周内被驳回次数 | ≥3次触发异常标记 |
| 断点率 | 流程中断/挂起实例占比 | >5%触发排查 |
| 异常集中度 | 异常是否集中在特定节点/人员/时段 | 集中度>60%需定向分析 |

## 可视化监控看板（HTML）输出规范

当用户提供流程轨迹数据时，**必须**生成交互式HTML监控看板文件。

### HTML看板必备组件

| 组件 | 内容要求 | 图表类型 |
|------|----------|----------|
| **核心指标卡片区** | 平均处理时长、超时率、异常率、一次通过率 | 数字卡片+趋势箭头 |
| **流程类型分布** | 各流程类型占比、数量 | 饼图或柱状图 |
| **业务域分布** | 各业务域流程数量 | 横向条形图 |
| **流程健康度排名** | 各流程的超时率、异常率对比 | 表格+进度条 |
| **节点瓶颈分析** | 各节点平均停留时长 | 横向条形图 |
| **异常预警列表** | 高风险流程、异常实例明细 | 表格+状态标签 |
| **SLA达成情况** | 各SLA等级的达成率 | 仪表盘或环形图 |

### HTML技术规范

- **框架**：使用原生HTML+CSS+JavaScript，不依赖外部框架
- **图表库**：使用ECharts或Chart.js CDN引入
- **样式**：响应式布局，适配不同屏幕尺寸
- **交互**：支持图表hover提示、数据筛选、时间范围切换
- **文件命名**：`[流程名称]_智能监控看板.html`

## 流程治理最佳实践（内置知识）

### 异常定义原则
1. **驳回异常**：同一流程实例一周内被驳回≥3次，定义为异常流程，需人工介入
2. **超时分级**：超时≤50%为黄色预警（催办），超时>100%为红色报警（升级处理）
3. **节点集中预警**：同一节点在同一时段异常数量>整体的60%，触发节点专项分析

### Owner收到的应该是报告，不是数字
- 错误方式：推送"超时率：38%"
- 正确方式：推送"合同审批超时率本周达38%，法务节点为主要卡点，建议授权二级法务处理标准合同"

### 优化决策优先级
1. 先排查规则问题（最容易改，成本最低）
2. 再看流程设计问题（需要重新建模）
3. 最后考虑执行者问题（需要培训）
4. 有条件时考虑自动化（高置信度场景）

## 反问用户的问题模板

当信息不足时，优先级顺序：

1. **流程基本信息**
   - "这个流程的核心业务目标是什么？（比如：3天内完成审批）"
   - "流程有哪几个主要节点？"

2. **当前痛点**
   - "目前这个流程最让人头疼的问题是什么？"
   - "有没有已知的卡点或高频问题？"

3. **Owner情况**
   - "谁是这个流程的负责人？他/她希望收到什么格式的报告？"

## 注意事项

1. **方案要可落地**：所有指标和阈值必须结合用户的实际业务场景，不能照搬模板数字
2. **Owner视角**：始终从流程Owner的角度设计报告，他/她收到的应该是"建议"而不是"数据"
3. **循序渐进**：建议先监控2-3个最关键指标，跑通之后再扩展，不要一次设计10个指标
4. **不混淆流程挖掘**：本Skill聚焦于设计监控规则，不做历史数据的回溯分析（那是process-mining的工作）
5. **必须有HTML输出**：当用户提供数据时，必须生成可视化HTML看板，不能只输出文字报告
